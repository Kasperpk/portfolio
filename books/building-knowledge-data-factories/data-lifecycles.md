Several inventions came along so that in 1966 IBM could introduce the very first database management systems or DBMS for short. The DBMS software is not only an abstracting over storing data on disks. It is also a system that enables interaction between the database and end users by introducing expression languages to locate, delete and in general manage the data in the database. Secondly it helps manage data by keeping track of what is stored in the database through a metadata store that is separate from the actual data stored. This metadata describes the data stored in the primary database both in terms of the relations and all the attributes which belong to that relation along with their data type. The DBMS software package introduces functionalities which distinguishes it from file-system processing by including features such as isolation and atomicity. These features makes sure that each multiple *transactions* (insert, update, retrieve) can be performed on the database at the same time. This allows multiple clients to interact with the database at the same time, something which is necessary for many business applications where many people use the applications. The database systems also comes with features such as indexing, caching and query optimizations that improves the user experience by allowing faster search of data on disks and faster query results. Additionally backups and data validations provides possibility for recovery of data. DBMS are complex pieces of software with many different components which allows all these features to coexists. ### The relational database model
The relational model is a very nice abstraction that presents to the user data in logical groupings called **relations** or tables. This is quite far from the physical implementation in hardware but is extremely logical for people and it is closely related to mathematical logic Codd grounded his ideas in set theory and first-order logic. A relation can be thought of as a *set* a mathematically concept which contains unique elements that share some characteristics. When the set is a relation in a database then each unique element represents a tuple or row that describe the attributes of one specific element. A field is then a specific cell that intersects a row and a column, something we often are familiar with from spreadsheets. For examples we may have the set $Customers$ with the tuple $(id, name, city)$. A single element could then be describes as $(1, Peter, Birmingham)$. The user can query the data in the relational database which we may express mathematically with relational algebra. For example we may write an operation as $\sigma_c(C)$. Which means select all customers where some condition $c$ is true. For example if we wanted to select all the customers from Italy we could write $\sigma_{country} = \space 'Italy' \space (Customers)$. As mentioned SQL is the primary language and to query data we define a `SELECT` statement that chooses a subset of the entire set or combination of sets which satisfies some condition. The above query could be written in SQL as `SELECT * FROM CUSTOMERS WHERE Country = 'Italy'`. There are many other operations we can do in the relational model which makes up the data query language (DQL) which is the part of SQL that contains the functionality to query data. Every such operation take one or more relation as input and produce a relation as output. The basic operations include selection as we have seen but also operations to select specific columns, combining two relations a single union or cartesian product and finding the difference between the relations corresponding to finding the set difference in mathematical logic. A projection written in relational algebra that selects the columns name and address attributes from the customer relation may look like this $\pi_{(N,A)}(C)$. In SQL `SELECT Name, Customer from Customers`. The union operation may combine two tables with the same or similar attributes. This is more often used in analytical databases where multiple sources often are combined. Nonetheless it can look like this $(CustomersNew) U (CustomersOld)$ where supposedly combine some new customer table with an old one. The operation in SQL uses the `UNION` command. Notice the * is used to signal that all columns are projected. 

```
SELECT * 
	FROM CustomersNew
UNION
SELECT *
	FROM CustomersOld
```

The set difference is used to find the elements of some relation that are not in another relation using the (-) sign. For example $(CustomersNew) - (CustomersOld)$ finds the new customers that are not also in the old customers table. In SQL the `EXCEPT` command is used to perform this operation. 

```
SELECT * 
	FROM CustomersNew
EXCEPT
SELECT *
	FROM CustomersOld
```

The cartesian product is an operation that combines every row in one relation with every row in another. There may be many uses for this when using and querying the data available in the database. The $X$ sign is used so two relations $C$ and $B$ are combined into a cartesian product mathematically expressed as $C \space X \space B$. Let $B$ be some relation `BankAccounts` then the SQL implementation is as follows using the `CROSS JOIN` command 

```
SELECT *
	FROM CustomersNew
CROSS JOIN
	BankAccounts
```

In addition to the basic operations there are also some derived operations which combine the basic ones to perform more complex ones. This includes intersection, division and different join types which are very useful particularly when it comes to making insights and answering questions of the data in the database. There are other parts of the SQL language such as the data definition language (DDL) used to create, alter and delete data elements, data manipulation language (DML) used to insert, update and delete data, data control language (DCL) used to grant and revoke access rights and permissions to objects in the database. Finally there is transaction control language (TCL) used to group a set of tasks into a single execution unit. Different types of users will use different languages. For example database administrators may use DDL, DCL and TCL while users of applications will use query options and manipulation often through the interface of their application. 

**Keeping integrity in the database**
One of the important concepts of the RDBMS is to enforce integrity. **Domain integrity** can be explained as the rules that ensures data is consistent and accurate. For example if the data type of some attribute is an integer, then new inserts into that table has to be an integer for that attribute. The second type of integrity is **entity integrity**. It's necessary that each row or tuple in a table is unique. This is ensured by creating a key for each row which much be unique and cannot contain NULL. A **primary key** is the column chosen for a specific table which hold the uniqueness and not null constraint. Sometimes it can be combination of columns that make up what's called a **composite key** and then there must be uniqueness for that combination of attributes. Thirdly we have to make sure that rules of relationships between tables are followed. In the RDBMS this is called **referential integrity**. Relationships between tables can be seen as parent-child relationships and when you want to link a parent to a child, you refer to the child by putting the primary key of it's parent in the child table as a **foreign key**. This integrity type is called and for example makes sure you can't delete a row from a parent table which references some child table.

**Normalizing data in the database**
When we work with databases we may take it for granted that the data is structured fairly good. In the relational database the concept of normalization ensures that data is possible to work with. In this example i will try to show how important the way we structure data is and how normalization is giving us clear rules to follow. Consider for example the example data in the table 3.5 below.

His normalization procedure in general follows a hierarchy of normal forms starting from 1NF before moving to subsequent forms. To adhere to first normal form we are going to eliminate those repeated groups. The  **second-normal form** (2NF) removes partial dependencies which means we split entities out so that the attributes depend only on the primary key of that table. That means we now have five separate relations closely mirroring our logical data model.

Now at this point the data is normalized also adhering to **third normal form** (3NF) which would remove any transitive dependencies for which there is no in this example. That would have been the case if there was a  non-key attribute that depended on another non-key attribute. For example if the birthdate of the customer was placed in the account table then that would have been a violation to the third-normal form. There exists more constraint forms of normalization but for the most part 3NF is the one that relational databases adhere to. The result of normalizing is data structured in a way that is much easier to maintain and work with. For example if some department change name or type, we only have to change it one places and all subsequent queries will show the change. Similarly if a new customer was created but no account was assigned to it yet, the one-big-table structure would have trouble representing that meaningfully. In general we can say that normalization doesn’t remove data — it removes unnecessary repetition by making dependencies explicit.

## Data warehouses and the data well
The introduction to this chapter highlighted multiple forces that led to the invention of the data warehouse namely complicated "spider-web" architectures and the ability to integrate data. However there is also a third reason. A DBMS system designed for operational applications also called online transactional processing (OLTP) has demands on them which are alleviated by for example introducing indexing algorithms. A typical operation is to insert, update or access a few rows which can be done efficiently with clever indexing and hash keys. It's also an important requirement in operational databases that the data is quickly altered or inserted so it reflects the current reality of the business immediately afterwards. Analytical queries on data such as "what was the total amount of sales last month?" requires retrieving larger subset of data and calculating aggregates, something that the clever indexing algorithms above do not solve particularly well. With analytics the need for having real-time updates of data is more a nice to have than a nice to have in most cases. For example if you look at sales over a month then the last 10 minutes of data does not weight as much as the past 30 days. Because the demands and data retrieval patterns are so different it was again the best option to separate transactional storage from analytical. When looking inside the analytical database we can spot this difference for example often in the way data is laid out. In a transactional database data will be stored row by row where all the attributes of a single row are next to each other. This is very nice when we want to retrieve or update a few rows because they are located close to each other on the disk. However in a analytical query we often decide on a few attributes but more rows such as transactions the last 30 days. Therefore analytical databases are often stored column-oriented for example storing the actual sales amount column in the same file. To make this work the order has to be preserved in the row so the 25th row in the one column file can be combined with 25th row in another file to assemble an entire row. (p. 97, Kleppmann) Popular file formats in analytical databases such as parquet and delta tables are using columnar storage format together with compression algorithms to optimize storage and query performance. 

### Subject oriented, integrated, non-volatile, time-invariant enterprise data
In came the data warehouse which Bill Inmon defined as the subject-oriented, integrated, non-volatile and time-invariant store of data. The data warehouse is an architecture and not a technology and in the early stages there where some debate about how it should look like. In 1996 Ralph Kimball published the book "*The Data Warehouse Toolkit: Practical Techniques for Dimensional Data Warehouses*". In the same period Bill Inmon published books on data warehousing such as the *Corporate Information Factory* in 1998 and *Building the Data Warehouse* in 1992. Inmon disagreed with Kimball on how a data warehouse should be designed. Kimball proposed the idea of making the data warehouse a collection of ready made data marts to begin with. Inmon argued that data instead should be landed inside the data warehouse first before transforming it further. His argument was that the requirements across departments and business units were so diverse that it should be possible to mold the same data in different ways to build data marts from the same set of raw materials. ### From data warehouse to data intelligence platforms and back
There has been several evolutions to the way of designing and implementing data warehouse solutions. Vendors like Microsoft, IBM and Oracle started offering software products like Microsoft SQL Server, Oracle Exadata or IBM Db2 Warehouse which became cornerstone in this area. Microsoft's SQL Server Integration Services (SSIS), Oracle’s Data Integrator (ODI) and IBM’s DataStage complemented the warehouse by allowing extraction of data from various sources and also transformations which together became known as ETL or extract-transform-load. With new capabilities within storage and compute new versions and technologies have been introduced to the market. The data lake term was introduced in the 2000s with Hadoop and MapReduce as the major technological drivers. Traditional data warehouse storage was robust at handling structured data and ingesting from relational databases. However as we have seen not all types of data fit that profile very well. The digital reality of businesses can exists in more unstructured formats that fit other types of databases such as document and graph databases. It may be files of text, sound, video and images or internet-of-things data something which the data lake could handle. Although the data lake with Hadoop and MapReduce won popularity it wasn't until the Spark project and the delta table storage format that we would see the next major overhaul of traditional data warehouses. One of the major innovations was the delta table an open-source columnar storage format that provides ACID transactional capabilities by keeping a transaction log. Secondly the Spark project introduced improvements to the existing distributed computing engines enabling data tasks to be distributed among several compute nodes. The founders of Spark introduced the term Data Lakehouse and Inmon in 2021 co-authored *Building the Data Lakehouse* which rests on the shoulders of the pioneering work he and Kimball and others did on data warehousing. This Lakehouse architecture evolution allows companies to take advantage of all the many advancements in compute and storage emerging from the ideas of the data lake while keeping data warehouse concepts intact. Secondly Lakehouses integrate support for machine learning and infused artificial intelligence allowing data scientist to enhance productivity. Thirdly, the data lakehouse platform allows different languages to query and manipulate data. This is also convenient as for some types of work Python may offer needed capabilities, while those more comfortable in SQL can use that to query and manipulate data.

### Data processing architectures
Once data is in the central repository it is in the hands of the data literate to make it such that raw data is combined and transformed for analytics. Following the advice of Bill Inmon many organizations follow the idea of having a landing zone and transform that same raw data for different sue cases. Early on when storage was very limited and expensive this may have met more resistance but gradually people started to have a two-tier architecture from where they could built data marts or (hyper)cubes which i will introduce soon to serve different analytical needs. The more recent version of this idea is the **medallion architecture**. The medallion architecture keep at least three versions of data. Some use the words bronze, silver and gold, others use raw curated and transformed and a third group something totally different. The names and numbers of layers are not the important thing, instead the concepts of creating a reliable data processing framework which can serve the needs of the organizations as a whole and individual needs of single departments, business units and teams. The medallion architecture represented In Matt Palmer's book *Understanding ETL* visualizes the typical process where multiple sources are ingested and stored in different curated versions through bronze, silver and gold layers.

![](Data%20flow%20in%20ETL.png)

As it may be visible it has many similarities with the data warehouse architecture and in many ways it's just a natural evolution reflecting changing demands and new technological advancements over the years. Piethein Strengholt the author of the book *Medallion architectures* discusses this architecture and provides some general guidelines. For example data in the bronze zone should be as Inmon would suggest touched as little as possible. As data moves through the data and knowledge centric factory it becomes refined often with some rough transformations in the silver layer where it's common to select, renaming and cast data types of columns. These transformations are typically so general that many data assets can be processed by the same jobs. In the gold layer as we shall see the data comes into the final pieces suitable for redeeming value from it. This is where each payload often receive dedicated processing as business logic is defined and data is harmonized across source systems. This is where in-depth business and data understanding becomes very valuable and where conversations with business people and analysts around requirements actually materialize. 

It's not to say that transforming data has to be done in this way, but in practice this has been working quite well for some time. These are some general guidelines but the most important thing is that the needs of the organization is met and there are options for different design choices based on specific needs and preferences.

### Dimensional modeling vocabulary
To participate in discussions about dimensional modeling we need to understand that there is a certain vocabulary needed. There are many similarities with entity-relationship diagramming is this model builds on top of that. For example the concept of primary keys and foreign keys functions exactly the same as in relational databases. Cardinality and relationships likewise. For example in the hypothetical example of an hospitality business above each room needs to be assigned a unique identifier so when a reservations that reservation can link to that room. The reservation also needs a date so we can link it to some date dimension. Now there are ways in which dimensional modeling differs from the models introduced earlier. One of those ways is that the dimensional model distinguish between tables that hold transactional data and more staple entity data. The transactional table is the one that populates the cells in the sketching figures above and the data is often numerical in nature, just like number of reservations. This type of table is called a **fact table** and often is the center of the model. Tables that surrounds the central fact table are called **dimensions** and describe the facts from different perspectives such as date, room type and so on. A simplistic sketch is shown in figure x.x and as you may agree it could look a bit like a star, why it's also referred to as a **star schema**.

![[Dimensional model|center]]

The fact table must contain foreign keys for all the dimensional tables it needs to link to, in order for the star joins to take place. In dimensional models this often introduces a third key type which is called the *surrogate key* used to link facts and dimensions. These are keys that are generated outside the operational systems that typically generate data and serves multiple purposes. For example the primary key may hold sensitive information like a social security number. This should perhaps not be visible for users. Secondly the primary key may consist of multiple columns some of which the data type is not optimal for processing. The surrogate key tackles this by creating a system constructed key, again a unique value for each row, but using random integers. This both hides the potentially sensitive information from the user but also allows the computer to work faster as they work better with integers than strings and date types.

Similarly our design of knowledge and data-centric solutions that helps monitor the stocks of the business can take departure in the conceptual data model. It's not always the case that organizations have their enterprise data model in place or use it at all. When we engage in conversations with stakeholders across the organization or ourselves it's a valuable skill to translate those conversations into the important stocks to monitor building a data model in your mind, later putting it down using nouns and verbs, entities and relationships to create a shared blueprint. A model may help create a shared view of the important entities (stocks) and their relationship or interconnections to use systems thinking language. 
### Evolutions of dimensional modeling
The concept of the data warehouse has survived the decades because it's technology agnostic and flexible enough to new evolutions. The same is true for dimensional modeling. Over time the dimensional modeling techniques have expanded. For example as expression languages like SQL and later DAX have over time introduced new capabilities to the dimensional model. DAX makes it possible to enrich models with operations to create custom measures for example to computer averages, maximums and so on over different attributes. There are an almost unlimited number of ways to utilize formulas available in these OLAP compatible languages, creating a whole professional discipline in mastering them. Later when visualization software became powerful enough dimensional models could be analyzed using graphs and diagrams, something we as humans are much stronger at. This has allowed the dimensional model to stay relevant to this day where it seems we mostly limited by lack of imagination and technical abilities. Complex and advanced dimensional modeling techniques such as bridge tables, slowly changing dimensions, hierarchies also have been developed in response to some analytical need. For example to increase performance, deal with difficult design problems such as many-to-many dimensions or very complicated data formats and structures. Once we master how to handle many-to-many relationships, hierarchies in the dimensions, develop custom measure and so on we find that this extendable multi-perspective cube is very potent and dynamic. Important we do not use the dimensional model just because it seems to have worked on other problems, but instead for the reason that it is the method that most effectively meets the information needs of the user. 
## Causal models, scientific methods and data mining
The dimensional model is particularly great at performing monitoring and descriptive analytics, however there are multiple forms of analytics some of which i described earlier. Model based analytics with a statistical or mathematical model is less about asking retrospective questions of data, but more about mining the data. In the very early days of decision support the word *data mining* emerged which described an alternative approach to using data. Instead of puzzling data together as dimensional star schemas, data mining explores data in a more scientific way. This exercise have grown in to the field of data science and *machine learning* which search for gold in data using statistical and mathematical methods.
