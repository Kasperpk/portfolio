**Running an instruction on the Von Neumann model**
Harris and Harris (2012) produce several examples of how the Von Neumann architecture may perform instructions as defined by the ISA, using the MIPS architecture as an example. The execution of a program or algorithm runs through several phases, controlled by the oscillating clock mentioned in the section on sequential logic. This phasing is also referred to as the *pipeline* of running instructions and can be expressed using the model of a **finite state machine**. A finite state machine shows how a process like executing a program progress, dividing it into well defined stages using arrows to show the relationship between them.

The first phase, called the *fetch* phase involves loading in the next instruction to be run into the special purpose register called the program counter or PC.

Figure 3.3 shows the instruction fetch stage, where on the clock rise, the address in the program counter fetches the instruction, here in some dedicated instruction memory segment. Once that is done, depending on the instruction, some bits may routed to be decoded by the control unit that set signals based on the operands of the instruction. Other bits may be directly supplied to the ALU or other units for example constant values. For example the load instruction that load some data to one of the registers. The data lies at an address that is equal to the base address residing in one of the registers plus some immediate constant value. Notice that througout the stages there are clock signals connected to all of the components. This allows the instruction cycle stages—fetch, decode, execute, and write-back— to be aligned and synchronzied at the rising clock edge (Harris and Harris 2012, 369). At this point the instruction is done executing and the machine can jump back to stage 0 and fetch the next instruction. What has been studied so far is how a single-cycle machine would complete clock cycles using 32-bit instructions. What is added here is the control signals that the control unit expose to the other components when decoding the instruction opcode. The following signals have been added. 
* RegWrite: Signal if the result should be written to a register
* RegDst: Multiplex between getting register address from instruction bits 15:11 or 20:16 depending on instruction type  
* ALUSrc: Multiplex between register or sign immidate as source operand for ALU
* ALUControl: Control bits for the ALU
* Branch: Signal would be 1 for operations like beq, that branch if ALU output is zero after subtracting two numbers
* MemWrite: Write enable signal to data memory
* MemtoReg: Multiplex between ALU result and data output from data memory

What should be a relief here is that, the units that make up this microarchicture have all been constructed in the previous section combinational or sequential logic. These in turn are built from elementary logic gates, which in turn are built from transistors. Thus we have now a fully working machine built step-by-step without cutting any corners. The single-cycle processor execute an instruction in one single cycle. This design have a few weaknesses as pointed out by Harris & Harris (page 381-382). Particularly they point out that: The clock cycle has to be long enough to support the slowest instruction even if most instructions could be executed much faster. Secondly there needs to be multiple adders (3), one for the ALU and two for the PC logic. Thirdly there single-cycle design had separation between instruction and data memory, something which introduces unnecessary complexity. To deal with these the authors introduce a multicycle microarchitecture, that breaks down instructions into smaller steps. With this design, some instructions can be performed in a small number of steps, while others take longer. Taking a look at their complete multicycle processor design in figure 3.8, there are really only a few differences. First of all the instruction and data memory has been combined, which is seen in all modern microarchitectures. This requires an additional control signal, *IorD* that handled the routing of data inputs and outputs from the two parts of memory. Secondly what noticeable are some intermediate registers, what Harris & Harris calls *nonarchitectural* registers. These hold values temporarily which allows instructions to be executed in multiple steps. Thirdly because there now is only a single adder, there has been added a multiplexer that also enables the ALU to handle PC logic. The logic for the PC is as follows: If there is no branch instruction, or if the branch is not taken, the PC needs to be incremented by a constant 4 since each instruction is 32-bits store in byte sized groupings such that first instruction for example is stored in memory addresses 0 to 4.

**Other ways of improving microarchitecture**
Computer engineers have battled many hours with ways to improve microarchitectures, to run programs faster and using less power. Several clever techniques have been introduced such as *pipelining*, *branch prediction* and *multicore processing*. 

**Pipelining** refers to the idea of letting the same processor run multiple instructions simultaneously. If we think about, once an add instruction for example has been loaded into the instruction register, it will in the next clock cycle run into the control unit and begin interacting with the register file. It no longer uses the memory and therefore the next instruction may be fetched. Once the add instruction is decoded and supplies it's operands to the ALU, and the result is stored in the ALUOut register, then the next instruction can be decoded. This is the idea of simultaneous execution and something that assembly lines in manufacturing companies try to take advantage of. It's like when you're boiling pasta, while listening to your favorite podcast, running the dishwasher and cutting out veggies for a salad. The timing diagram in figure 1.11 shows this exact idea. As soon as the previous instruction is done with stage 1, the next instruction can move into that stage increasing the throughout put by a lot. Although there is extra latency introduced due to additional combinational logic in the form of pipeline register, this 5-stage multipipeline processor still increase the throughout put by a factor of four compared to a single-cycle version.

Obviously there may be problems if the next instruction is dependent on the result of previous instructions. That may be true if you have a program like the code example 1.6.

There general term for this is pipeline *hazards*. Without going into detail this challenge can be solved using *forwarding* or *stalls* thus adding extra complexity to the microarchitecture. Undefined instructions and arithmetic overflow also poses challenges to hardware designers. These two cases are typically handled with **exceptions**. These are some of the details that must be kept in mind when trying to optimize processors or the other elements of a Von Neumann model. 

**Modern microarchitectures** are exceptionally fast and advanced. The 5-stage pipeline is more around 10 to 20 stages in current technology, referred to as *deep pipelines* (Harris & Harris, page 435). It also uses a technique called *branch predictions*, which is making educated guesses whether the branch is taken or not and according to (Harris & Harris page 438) good branch predictors reach accuracies of up to 90%. In the mid 2000s, the microarchitectures begun to introduce *multicore* processing which is essentially multiple processing units on the same chip. This enables execution of multiple programs simultaneously also known as *multithreading*, something which was also possible before using superscalars. It also enables running the same program faster by splitting it up between more cores, however this is much more challenging. Since programs often have many dependencies, there are limits to how much improvement can be achieved by this exercise. Harris & Harris claims that the number of instructions that can be executed simultaneously rarely exceeds 2 or 3 even if you have 5,6,7 or 12 cores available. (Harris & Harris page 443).
